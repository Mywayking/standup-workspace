# 喜剧分析工具 - 重新设计

## 核心理念

**一个演员拿起手机/打开电脑，唯一想知道的是：**
> "我这段子行不行？哪里不行？怎么改？"

所有其他概念（项目、阶段、导入导出）都是噪音。

---

## 新设计：单页写稿台

### 布局（极简两栏）

```
┌─────────────────────────────────────────────────────────┐
│  🎤 喜剧写稿台                              [⌘K 素材库]  │
├──────────────────────────┬──────────────────────────────┤
│                          │                              │
│  【输入区】               │  【反馈区】                    │
│  粘贴或输入你的段子...    │                              │
│                          │  结构：铺垫→递进→callback     │
│  至少20字                 │  技巧：观察式喜剧、callback   │
│                          │  态度：讽刺、略带无奈          │
│  [分析这段]  ← 主按钮    │  问题：第三段节奏拖沓         │
│                          │  亮点：callback用得巧         │
│                          │                              │
│                          │  相似素材 (2条)               │
│                          │  → 王十七: 医疗段子           │
│                          │  → 豆豆: 内卷话题             │
└──────────────────────────┴──────────────────────────────┘
```

### 核心交互原则

1. **粘贴就分析** — 不需要"上传"这个动作，粘贴完自动分析
2. **实时反馈** — 不需要等进度条，反馈直接出现在右侧
3. **素材库随手查** — ⌘K 弹出素材搜索，不需要离开当前页面
4. **没有项目管理** — 不需要新建项目，所有分析记录自动保存
5. **没有分段概念** — 用户只看到整体反馈和每段落的简要标注

### 极简模式（当前版本 → 新版本）

| 当前 | 重新设计 |
|------|---------|
| Dashboard + Workspace + KB（3个页面） | 单一写稿页面（1个页面） |
| Project → Script → Segment（3层） | 文本 → 分析结果（1层） |
| 上传 → 等进度 → 看结果（3步） | 粘贴 → 即时反馈（1步） |
| 分段对用户可见 | 分段对用户隐藏（后台自动处理） |
| KB单独页面 | ⌘K弹出搜索，不打断写稿 |
| 导出功能独立 | 按需导出（复制/下载） |

## 新后端架构

### 问题：mmx CLI 持续 SIGABRT

**解决方案：直接用 HTTP API 调用 MiniMax/DeepSeek/GLM**

不通过 CLI 包装，通过 curl/httpx 直接调用 API，好处：
- 绕过 CLI 的 SIGABRT 问题
- 支持流式输出（用户看到逐步反馈）
- 多 provider 兜底（DeepSeek → GLM → MiniMax）
- 部署简单（不需要安装 mmx CLI）

### API 设计

**POST /analyze** （核心端点）
```json
Request:
{
  "text": "段子全文",
  "mode": "quick | deep",  // quick=3秒反馈，deep=完整分析
  "want_references": true   // 是否返回相似素材
}

Response (流式):
{"type":"chunk","text":"正在分析结构..."}
{"type":"result","structure":"铺垫→递进→callback","techniques":["观察","callback"],"score":0.72}
{"type":"reference","segments":[...]}  // 可选
```

**GET /kb?q=内卷&technique=callback** （素材库搜索）
```json
Response:
{
  "total": 24,
  "results": [
    {"text":"...", "actor":"何广智", "structure":"callback", "tags":["内卷","职场"]},
    ...
  ]
}
```

## 新前端页面

只有一个页面：`/write`

```
/write          ← 主页面（写稿台）
/kb             ← 素材库（单独的，不是页面，是⌘K弹窗）
```

**URL 只有 /write，不再有 /workspace/[id]、/projects、/dashboard**

历史记录在右侧边栏（可收起），显示最近10次分析。

---

## 迁移策略

**Phase 1（最小MVP）：**
- 新建 `/write` 页面（写稿台）
- 保留现有后端 API，包装成新接口
- 保留 `/kb` 页面（但从⌘K也可访问）
- 旧版 Dashboard 暂时保留，用户可以逐步迁移

**Phase 2：**
- 历史记录功能（自动保存到 localStorage，后端同步）
- 多 provider 切换（DeepSeek 主，MiniMax 备）

**Phase 3：**
- 删除旧版页面（workspace、projects、dashboard）

## 评估

**优点：**
- ✅ 用户从"3步操作"变成"1步操作"
- ✅ 不需要理解任何技术概念（无Project、无Segment、无Step）
- ✅ 素材库随手可查，不打断写稿心流
- ✅ 直接 HTTP API，绕过 mmx CLI 的 SIGABRT
- ✅ 单页设计，代码维护成本低

**风险：**
- ⚠️ 需要重新写前端（投入较大）
- ⚠️ 历史记录需要后端支持
- ⚠️ 素材库⌘K弹窗体验需要调优

**建议优先级：**
1. 先做 Phase 1（单页写稿台），用户立刻感受到价值
2. 同时修复 mmx CLI 问题（切 HTTP API）
3. Phase 2/3 看用户反馈决定
