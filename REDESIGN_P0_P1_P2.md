# standup-workspace UI 优化技术方案

**版本**：v1.1（基于 v1.0.0）
**日期**：2026-04-16
**状态**：进行中

---

## 一、P0 首屏转化改造

### 1.1 首屏标题与副标题改造

**现状**：无清晰标题，左上角产品名 + 空状态
**改动**：

```
主标题（h1）：把段子贴进来，看看它到底卡在哪
副标题（p）：从结构、笑点、节奏、情绪推进、结尾几个角度帮你定位问题
```

**测试用例**：
1. 页面刷新后标题正确渲染，中文字体大小 ≥ 28px
2. 副标题正确显示，字号小于标题，颜色为 gray-500
3. 在 375px 窄屏下标题不断行（word-break 处理）

---

### 1.2 CTA 按钮视觉强化

**现状**："分析这段" 按钮视觉权重偏弱，和辅助元素视觉等级接近
**改动**：
- 主按钮背景从灰色改为 indigo-600 hover → indigo-700
- 字号增大至 16px，padding 扩大
- 按钮下方增加微文案："输入至少 20 字即可开始"

**测试用例**：
1. 主按钮背景色为 `#4f46e5`（indigo-600），hover 变 `#4338ca`（indigo-700）
2. 按钮文字"开始诊断"，字号 16px，加粗
3. 按钮 disabled 状态（输入为空）背景变灰，cursor: not-allowed

---

### 1.3 输入框 placeholder 升级

**现状**：`placeholder="输入段子内容..."`
**改动**：
```typescript
placeholder="把你不确定好不好笑的段子贴进来"
```

**测试用例**：
1. placeholder 文案正确显示为"把你不确定好不好笑的段子贴进来"
2. focus 时 placeholder 文字消失，不出现残留
3. 输入文字后 placeholder 完全消失，不出现残影

---

### 1.4 输入区顶部引导文案

**现状**：输入框上方无引导
**改动**：在输入框上方增加一行引导：
```
把你最不确定的段子贴进来，我们先找出最该改的地方
```

**测试用例**：
1. 引导文案在输入框上方，字体 14px，颜色 gray-500
2. 引导文案与输入框间距 8px
3. 在各种屏幕宽度下文案不断行

---

### 1.5 示例段子一键填充

**现状**：空白页没有示例入口
**改动**：在引导文案下方增加示例按钮：
```
[🔍 试试示例段子]
```

点击后自动填充一段完整段子（~200字），并滚动到分析按钮可见位置

**示例段子内容**（预置 3 条，每次随机选 1 条）：
1. 相亲段子（关于写bug）
2. 迟到段子（时间管理）
3. 地铁偷听段子（安徽大姐歧视理论）

**测试用例**：
1. 点击"试试示例段子"按钮，输入框自动填入内容（>50字）
2. 填充后页面自动滚动，分析按钮进入视口
3. 连续点击示例按钮，内容不重复累加（每次替换而非追加）

---

### 1.6 右侧空状态改造为结果预览卡片

**现状**：
```
⚠ 左侧输入段子后点击分析
评分、结构、技巧分析马上呈现
```

**改动**：右侧空状态改为"结果预览区"，展示未来结果的结构骨架：
```
你将获得：

✓ 一句话总诊断
✓ 最该优先修改的 3 个问题
✓ 已经有效的笑点
✓ 可直接替换的改写建议

分析完成后自动呈现
```

**测试用例**：
1. 右侧空状态显示为预览卡片样式（非纯文本）
2. 4个预览项以列表形式展示，带 ✓ 图标
3. 在分析完成后，右侧内容无缝切换为真实结果（无白屏闪烁）

## 二、P1 结果页行动性改造

### 2.1 结果页顶部：增加"一句话总诊断"

**现状**：结果页直接从 8 维评分开始，用户没有整体认知
**改动**：在评分卡片上方增加一行：

```
📌 总诊断（从所有评分和段落分析中提炼的核心结论）
正文：这段最大的问题不是观点不行，而是...
```

**样式**：背景 indigo-50，左侧蓝色竖线，字体稍大（16px）

**测试用例**：
1. 分析完成后，页面顶部正确显示总诊断内容（>10字）
2. 总诊断文字有蓝色左边框背景，区分于其他内容块
3. 总诊断内容随不同段子变化，不出现固定文案

---

### 2.2 结果页增加"最该优先修改的 3 个问题"

**现状**：8维评分全部平铺，用户不知道先改哪
**改动**：在总诊断下方、评分卡片上方增加"优先级问题"模块：

```
🔥 优先改这3个（从结构/技巧/节奏分析中提炼）

1. [问题标题] — [一句话说明问题]
2. [问题标题] — [一句话说明问题]
3. [问题标题] — [一句话说明问题]
```

**数据来源**：从 segments[].problem 字段提取 + 后端新增 top_problems 字段

**测试用例**：
1. 始终显示3条（不够3条时显示实际条数），每条有 emoji 和问题描述
2. 问题描述文字不超过 30 字
3. 如果 segments 返回的问题不足3个，模块高度自适应，不留空白

---

### 2.3 结果页增加"有效笑点"模块

**现状**：只显示问题，没有肯定做得好的地方
**改动**：在评分下方增加"有效笑点"模块：

```
✅ 已经有效的笑点

• "原文中已成立的优秀表达"
• "节奏感好的段落"
• "有潜力的结尾"
```

**数据来源**：从 segments 中提取 structure="笑点成立" 或评分 ≥ 4 的段落

**测试用例**：
1. 如果没有高评分段落，模块不显示（而不是显示空列表）
2. 每个有效笑点以"• "列表形式展示
3. 有效笑点最多显示 3 条（取评分最高的）

---

### 2.4 结果页增加"改写建议"模块（直接可用的）

**现状**：script_changes 以代码风格展示
**改动**：在评分区域下方增加"改写建议"卡片，格式：

```
✏️ 直接替换建议

[原句] → [优化后句]
位置：铺垫段第2句
技巧：增加预期违背

[原句] → [优化后句]
位置：结尾段
技巧：压缩节奏
```

**样式**：左侧橙色边框卡片，原句灰底，可点击复制

**测试用例**：
1. script_changes 每条以"原句 → 改写句"形式展示
2. 点击"复制"按钮，原文复制到剪贴板，有 toast 提示"已复制"
3. 如果没有 script_changes，模块不显示

---

### 2.5 结果页增加"下一步动作"按钮组

**现状**：分析完成后没有下一步引导
**改动**：在结果页底部增加按钮组：

```
继续创作：
[强化结尾] [增加包袱] [转成综艺版] [保存到素材库]
```

**按钮样式**：outline 风格，次级视觉权重，颜色 indigo-500

**测试用例**：
1. 4个按钮正确排列，文字对应功能
2. 点击"保存到素材库"弹出保存弹窗（含标签选择）
3. 点击其他按钮（强化结尾/增加包袱）在当前页面触发对应操作

---

## 三、P2 历史与素材库（简化版）

### 3.1 顶部导航增加"历史"入口

**现状**：无历史入口
**改动**：顶部工具栏增加历史图标按钮，右侧显示"最近分析 N 条"

**测试用例**：
1. 点击历史图标展开历史列表面板（右侧滑出）
2. 列表显示最近 20 条，每条显示：时间 + 段子摘要（20字）+ 分析结果一句话
3. 点击列表项回到对应分析结果

---

### 3.2 本次会话历史（内存级）

**现状**：刷新页面后历史丢失
**改动**：使用 localStorage 存储本次会话历史，最多存 50 条

**数据结构**：
```typescript
interface HistoryItem {
  id: string;
  time: number; // timestamp
  text: string; // 原文前50字
  summary: string; // AI总结的一句话
  result: AnalyzeResult;
}
```

**测试用例**：
1. 分析完成后，自动保存到 localStorage，key = `standup_history`
2. 刷新页面后历史列表仍然存在
3. 超过50条时自动删除最老的条目

## 四、技术依赖与接口变更

### 4.1 后端新增字段（可选，快速迭代）

如后端返回数据中增加以下字段，前端展示效果更好：

```typescript
interface AnalyzeResult {
  // 新增
  top_problems: string[];  // ["开场铺垫太长", "结尾收得太快", "转折不够自然"]
  effective_jokes: string[]; // ["这段的自嘲很成立"]
  overall_verdict: string;  // "最大的问题是前半段铺垫太久，建议压缩到3句话内"
}
```

**前端兼容**：如果后端暂无这些字段，前端从现有数据中计算（从 segments 提取 / 截取 evaluation 第一条作为总诊断）
